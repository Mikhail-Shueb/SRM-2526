# Step 2: Direct Kinematics — Beginner Guide

## What is this step about?

**Direct Kinematics** answers: *"If each joint is at angle q1, q2, …, q7, where exactly is the end-effector (the robot's hand) in 3D space?"*

This step takes the D-H table from Step 1 and builds a **Simulink block** that can answer this question automatically for any joint angles you feed into it.

---

## The Core Idea: Transformation Matrices

Each joint in the robot "transforms" space — it rotates and offsets the coordinate system.  
The D-H convention gives us a **4×4 matrix** for each joint:

```
A_i = [ R_i   p_i ]    ← R_i is a 3×3 rotation, p_i is a 3×1 position
      [  0     1  ]
```

To find where the end-effector is, we **multiply all 7 matrices in order**:

```
T = A1 × A2 × A3 × A4 × A5 × A6 × A7
```

This final matrix `T` is also a 4×4 matrix with the same structure:

```
T = [ R   p ]    ← R tells you the orientation, p tells you the position
    [ 0   1 ]
```

- **`p` (3×1 vector):** The `(x, y, z)` position of the end-effector in the world.
- **`R` (3×3 matrix):** The orientation (which way the hand is pointing).

---

## Why is it Symbolic?

Instead of computing `T` for one specific set of angles, we keep `q1`…`q7` as **symbolic variables** (unknowns). This means `T` becomes a formula in terms of q1…q7.

When we later want to know the position for specific angles (e.g. q1=0.5, q2=1.2, …), we just substitute those numbers into the formula — no need to redo the full calculation.

> **Analogy:** It's like writing the formula `y = x²` once, rather than computing the answer separately for x=1, x=2, x=3, etc.

This is what `DKin(KukaLBR())` does — it returns the symbolic formula for `T`.

---

## What is the Simulink Library?

The symbolic formula for `T` contains trigonometric functions (sin, cos) of q1…q7.  
We use `matlabFunctionBlock()` to automatically convert this into an **optimised MATLAB function** embedded inside a **Simulink block**.

This block:
- Takes **7 inputs**: q1, q2, q3, q4, q5, q6, q7 (joint angles in radians)
- Gives **2 outputs**: `Kuka_R` (3×3 rotation) and `Kuka_p` (3×1 position)

You can then drag this block into any Simulink model and connect it to other blocks.

## Understanding the Validation Tests

The validation works by plugging in joint angles where we **already know** what the answer should be, and checking if the model agrees.

### Test 1 — All joints at 0° (Home position)
- **Expected position:** `p = [0, 0, 0.926]` metres  
- Why? The arm is fully stretched upward. Since we set the base frame to the shoulder (`d1=0`), the total height is just the sum of the remaining links:  
  `d3 + d5 + d7 = 0.400 + 0.400 + 0.126 = 0.926 m` ✅
- **Expected orientation:** Identity matrix (no rotation — the hand points straight up) ✅

### Test 2 — Joint 2 at 90°
- Due to the twist (`alpha = pi/2`) on joint 1, the axis of joint 2 is aligned with the Y-axis. Rotating joint 2 bends the arm towards the negative X-axis.
- The arm above joint 2 has length `d3 + d5 + d7 = 0.926 m`.
- **Expected:** Hand moves out to `x = -0.926 m`, stays at height `z = 0 m` ✅

### Test 3 — Joint 2 at 90°, Joint 4 at −90° (L-shape)
- The arm first bends along the -X axis (joint 2), placing the elbow at `x = -0.400 m`.
- Because the local frame's Z-axis is now pointing sideways, rotating joint 4 by -90° bends the lower arm *down* along the -Z axis.
- **Expected:** Hand at `x = -0.400 m`, `z = -0.526 m` ✅

---

## What if the numbers are wrong?

| Problem | Likely cause |
|---------|-------------|
| Home position height ≠ 0.926 m | Wrong `d` values in `KukaLBR.m` |
| Home R ≠ Identity matrix | Wrong `alpha` values in `KukaLBR.m` |
| Test 2 end-effector didn't move outward | Wrong joint order or wrong signs on `alpha` |

## Key Takeaway

Direct Kinematics maps **joint angles → end-effector pose**. It is the foundation for all other steps:
- Step 3 (Inverse Kinematics) uses it to **validate** its results.
- Step 4 (Jacobian) differentiates it to get velocities.
- Step 5 (CLIK) uses it inside a **feedback loop**.
